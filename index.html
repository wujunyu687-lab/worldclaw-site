WorldClaw Overview

Overview of the WorldClaw execution graph: platform-facing access surfaces route requests into an orchestrator, which delegates work to specialized internal bots backed by a shared runtime and memory layer.

Unified Access Layer
Terminal, WeChat, and Feishu are treated as entry surfaces rather than as separate execution silos, so the same project can be operated across different interfaces without duplicating task logic.
Specialized Internal Bots
Instead of overloading one general chat bot, WorldClaw routes work to dedicated components for training, evaluation, paper search, and reporting.
Shared Runtime and Memory
Outputs, runtime state, and project memory remain attached to the same workspace, making long-running project work easier to organize and continue.

Abstract

WorldClaw is designed as a unified workspace for project execution rather than as a single-purpose repository. Its core objective is to separate platform access, orchestration, and task execution, so that training, evaluation, research search, and reporting can coexist in one structured environment without collapsing into a monolithic chat interface. The system connects terminal, WeChat, and Feishu entry points to a shared execution layer, where specialized bots take responsibility for distinct workflows. This design provides clearer task boundaries, more maintainable routing logic, and a more reusable project structure than directly wiring every workflow into a single front-end bot.

Core Components

TrainBot

A training control plane built around targets, variants, experiments, ledgers, and lineage rather than around one fixed repository layout.

  • Target and experiment contracts
  • Variant registry with aliases and inheritance
  • Resolved-config, summaries, bundles, and lineage
  • Smoke runs and comparison utilities

EvalBot

A benchmark-oriented execution layer that starts with entry selection and prechecks before surfacing environments, weights, execution steps, and output locations.

  • Benchmark entry registry
  • Environment validation
  • Weight readiness checks
  • Result location reporting

PaperSearch

A lightweight research-operations layer for topic search, proceedings crawling, and turning raw results into normalized project outputs.

  • Topic search
  • Conference proceedings crawl
  • CSV normalization and merge tooling
  • Search outputs organized for downstream use

MetaBot Bridge

The bridge layer handles IM integration, local API exposure, session routing, and runtime coordination without absorbing all business logic into the platform layer.

  • WeChat and Feishu integration
  • Terminal-facing entry workflow
  • Bot registry and routing
  • Shared runtime and memory coordination

Operational Workflow

1. Define structured project contracts
Targets, variants, experiments, and benchmark entries turn operational intent into explicit contracts instead of loosely specified prompts.
2. Route requests through entry surfaces
Terminal, WeChat, and Feishu all feed into the same execution graph, which keeps platform access separated from the task layer.
3. Delegate to specialized bots
The orchestrator forwards work to TrainBot, EvalBot, PaperSearch, or ReportBot depending on the structure of the request.
4. Persist outputs, state, and memory
Runtime artifacts, ledgers, summaries, and memory records stay attached to the same workspace, supporting long-running project continuity.

Public Site Scope

This repository hosts the public-facing project page only. It is intentionally separated from the implementation repository so the system architecture, workflow, and project framing can be shared without exposing the working codebase, credentials, or runtime configuration.
