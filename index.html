Abstract

WorldClaw is designed as a unified workspace for project execution rather than as a single-purpose repository. It connects terminal, WeChat, and Feishu entry surfaces to a shared execution layer, where specialized bots handle training control, benchmark evaluation, paper search, and project reporting.

The central design goal is separation of concerns. Platform access is handled by lightweight entry bots, while the actual task logic is executed by dedicated internal modules. This makes the system easier to maintain, easier to extend, and much less chaotic than pushing every workflow into one monolithic chat-facing bot.

Core Components

TrainBot

A general training control plane built around targets, variants, experiments, ledgers, and lineage, instead of binding the whole system to one training repository.

  • Target and experiment contracts
  • Variant registry with inheritance and aliases
  • Resolved-config persistence and lineage capture
  • Smoke runs, summaries, ledgers, and comparison utilities

EvalBot

A benchmark-oriented execution layer that begins with entry selection and prechecks, then surfaces missing environments, required weights, execution steps, and saved results.

  • Benchmark entry registry
  • Environment and dependency validation
  • Weight readiness and path checks
  • Output location and result reporting

PaperSearch

A lightweight research-operations layer for collecting recent work, crawling proceedings, and converting raw results into normalized outputs that are easier to reuse downstream.

  • Topic search
  • Conference proceedings crawl
  • CSV normalization and merge tools
  • Project-ready output organization

MetaBot Bridge

The bridge layer handles IM integration, API exposure, and runtime coordination. Its role is to connect the platforms to the task layer, not to absorb all business logic itself.

  • WeChat and Feishu integration
  • Local terminal-facing API
  • Bot registry and routing
  • Runtime and memory service coordination

Operational Workflow

01

Define the project contract

Training and evaluation work are first turned into explicit targets, variants, and benchmark entries, which reduces ambiguity and makes the pipeline auditable.

02

Route user requests through entry surfaces

Terminal, WeChat, and Feishu all feed into the same execution layer, so the project can be operated from different platforms without duplicating the core logic.

03

Execute specialized workflows

The orchestrator delegates training, evaluation, paper search, and reporting to the corresponding bot instead of treating all tasks as one generic conversation loop.

04

Persist outputs and memory

Summaries, bundles, ledgers, output paths, and memory records stay connected to the same workspace, which makes iterative project work much easier to continue.

Public Site Scope

This repository hosts the public-facing project page only. It is intentionally separated from the implementation repository so the architecture, workflow, and public overview can be shared without exposing the working codebase, credentials, or project-specific runtime configuration.
